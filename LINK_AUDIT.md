## Suspicious links and structural concerns

1. **No obvious module-to-root back link was confirmed in the sampled module headers.**
   - The root portal links to the modules.
   - Module pages appear to provide internal chapter/navigation controls.
   - A consistent visible back/home link to `../index.html` should be considered if users are expected to return to the main portal from a module page.

2. **External Google Fonts dependency.**
   - Module pages use Google Fonts via external `href` references.
   - This is acceptable for public web deployment, but it is still an external runtime dependency.
   - If the platform must work offline or in restricted client environments, fonts should be self-hosted or safely degraded.

3. **Inline CSS and JavaScript across many files.**
   - No missing standalone CSS/JS files were identified because the files appear self-contained.
   - The tradeoff is maintainability: fixes to quiz behavior, navigation behavior, branding, or layout must be repeated across many HTML files unless centralized.

4. **Root landing page link map is embedded in inline JavaScript.**
   - Current links are correct.
   - Future file renames can break the portal unless link auditing is automated.

5. **Historical renamed Introduction files could still be referenced internally.**
   - Older filenames that no longer exist include:
     - `Introduction/01_intro_cybersecurity.html`
     - `Introduction/05_intro_social_engineering.html`
     - `Introduction/07_intro_email_security.html`
     - `Introduction/08_intro_data_protection.html`
     - `Introduction/09_intro_network_security.html`
     - `Introduction/10_intro_internet_security.html`
   - No current root landing-page reference points to these old paths.
   - A deeper full-text audit should confirm that no module body still links to these old names.

## Recommended fixes

1. Add a visible, consistent module-level back/home link to `../index.html` if users should return to the portal after completing a module.
2. Add an automated link audit script that parses `href` and `src` references across all HTML files and fails on missing repository-local targets.
3. Add a CI workflow or documented manual command to run the link audit before any deployment.
4. Consider extracting shared CSS and JavaScript into versioned files such as:
   - `assets/css/training.css`
   - `assets/js/training.js`
   This should be done only after the current module behavior is fully stable.
5. Decide whether Google Fonts should remain external or be replaced with a local/self-hosted font strategy.
6. Run a deeper full-text search for historical filenames listed in the suspicious-links section and remove any stale references.
7. Add a case-study link audit for `real-life-events/` as a separate pass, because that folder was outside this requested scope.

## Audit limitations

The available GitHub connector in this session did not provide a direct full repository tree-listing command. The audit used repository metadata, known module inventory, direct file fetches, asset verification, and repository compare output. Because several HTML files are large and contain inline minified JavaScript/CSS, a future automated local-clone audit is still recommended for exhaustive line-by-line validation.
